# 项目公共上下文：RT-DETR + MDVSC

本文档是仓库的共享设计上下文，是当前目标系统、模块边界、建模假设与训练意图的统一参考。

## 项目目标

项目目标是构建一个联合视频语义通信与目标检测系统。

发送端接收一段视频片段，提取 RT-DETR encoder 侧的多尺度语义特征，通过 MDVSC 风格的语义通信模型对其进行压缩与传输；接收端恢复语义表示，并服务于两个下游任务：

- RT-DETR decoder 与检测头的目标检测。
- 帧级视觉重建支路。

优化目标既不是只做检测，也不是只做重建，而是在通信约束下实现联合最优。

## 范围区分

仓库中目前存在两个层面的“真实定义”：

1. 当前实现状态。
   当前仓库已经有单图 RT-DETR 基线和一个最小语义通信 MVP，用于验证接口、切点和信道仿真。

2. 目标项目架构。
   最终项目目标是视频输入、latent 域语义压缩、联合检测与重建，以及基于 MDVSC 的通信建模。

当前代码库的作用，是逐步逼近这个目标架构。临时的单图实验是接口验证工具，不代表最终系统定义。

## 已达成一致的系统视图

当前约定的目标管线如下：

1. 输入视频片段或 GOP。
2. 对每一帧运行 RT-DETR backbone，并取 encoder_input_proj 之后的三层 256 通道 projected backbone 特征。
3. 在 projected backbone 特征切点收集多尺度语义包。
4. 通过语义 adaptor 和 latent compressor 将原始 RT-DETR 特征转成更紧凑的 latent 域表示。
5. temporal aggregator 从 latent 序列中提取时间共享语义。
6. 将 latent 序列分解为公共分支和个体分支。
7. 语义通信模型施加重要性建模、码率控制和信道感知传输。
8. 接收端恢复 latent 语义特征，经过 per-level refinement 后还原为与 RT-DETR 兼容的多尺度特征图。
9. 恢复后的语义特征送入两个头部：
   检测分支，用于 RT-DETR decoder 和 detection head；
   重建分支，用于视频帧重建。

## 切点与传输对象

RT-DETR 切点固定在 backbone 输出经过 encoder_input_proj 投影到 256 通道之后、hybrid encoder 之前。

传输对象不是 RGB 视频，也不是 decoder query。传输对象是 encoder 侧的多尺度语义特征包。

对当前 RT-DETR 参考模型，这个特征包包含三层：

- level 0，stride 8；
- level 1，stride 16；
- level 2，stride 32。

原始 feature bundle 是语义源，但真正的传输不应直接发生在原始 256 通道特征图上，而应在 latent 压缩之后进行。

## 输入假设

目标输入始终是视频，而不是单张图像。

当前统一的张量视图为：

- 视频输入：`[B, T, 3, H, W]`
- 每个 level 的 encoder 特征序列：`[B, T, C_l, H_l, W_l]`

单图实验仍然对 shape 契约和信道效应调试有用，但不定义最终模型语义。

## 项目版 MDVSC 的设计方向

仓库中的项目版 MDVSC 应继承现有 MDVSC 代码中的关键思想，同时把工作域从 RGB 帧重建改造成语义特征通信。

项目版 MDVSC 需要保留这些思想：

- 公共与个体分解；
- 重要性感知的掩码或码率控制；
- 信道感知的 latent 传输；
- 轻量残差修复。

项目版 MDVSC 需要改变这些前提：

- 处理对象从 RGB 像素改成 latent 语义特征；
- 输入从独立图像改成视频特征序列；
- 输出从单一路径改成同时服务检测与重建两个头部的恢复语义图。

## Latent 域压缩策略

已达成一致的压缩顺序是：

1. 对原始 RT-DETR 特征做适配；
2. 将其压缩到 latent 域表示；
3. 再决定各个分支是稠密传输还是稀疏传输。

稀疏传输决策应在 latent 域中完成，而不是直接在原始多尺度特征图上完成。

当前的默认设计偏好是：

- 公共分支：更小、更稳定，通常采用稠密传输或弱稀疏；
- 个体分支：更适合做结构化稀疏传输。

推荐的稀疏单元应是结构化的，而不是完全非结构化的。可选形式包括：

- 通道；
- 空间块；
- 通道与空间块的组合。

完全非结构化的逐元素 top-k 掩码可用于探索性分析，但不应作为最终传输格式，因为索引开销可能反而主导整体码率。

## Temporal aggregator

temporal aggregator 是把时间维上的 latent 序列汇聚成共享时间语义表示的模块。

对于每个被选中的 level，它接收逐帧 latent 序列，并输出该 level 的时间公共表示。每帧的个体残差语义则相对这个公共表示来定义。

第一版实现应优先采用轻量的 temporal aggregator，例如时间平均加局部修复，或一个小型 temporal convolution，而不是重型 attention 堆栈。

## Per-level refinement

语义解码之后，每个恢复出的 level 都应先经过一个轻量的 per-level refinement block，再被下游使用。

per-level refinement block 的作用，是修复信道扰动、mask 引入的伪影以及公共/个体融合误差，同时保持该 level 的原始张量形状不变。

这个修复步骤位于恢复语义包被检测分支和重建分支共同消费之前。

## 接收端双头设计

接收端有两个共享恢复语义包的下游头部。

1. 检测头。
   恢复出的多尺度特征回送到 RT-DETR decoder 和检测头。

2. 重建头。
   一个独立的 semantic-to-image reconstruction decoder 从恢复语义表示中重建视频帧。

如果项目目标是检测与重建联合最优，那么重建分支就是系统定义的一部分，不能被视为可有可无。

## 训练目标

目标项目采用联合目标，而不是单任务目标。

预期的损失组成包括：

- 检测损失；
- 重建损失；
- 语义特征一致性损失；
- 码率或稀疏性正则项；
- 可选的时序一致性损失。

重建指标必须保留在训练和评测中。检测不会取代重建；项目目标是在码率约束下实现联合最优。

在训练策略上，项目默认采用“先建立无语义通信的双头任务上界，再训练通信模块逼近该上界”的思路，而不是一开始就把通信模型和双任务目标一起端到端训练。

这个上界模型的作用是：

- 先验证 RT-DETR encoder 语义特征本身是否足以同时支撑检测和重建；
- 给后续 MDVSC 训练提供稳定的任务参考分布；
- 把“任务模型能力不足”和“通信压缩损失过大”这两类问题拆开。

## 推荐训练阶段

推荐采用分阶段训练路径：

1. 无语义通信的双头上界模型训练。
   输入仍然是视频，但发送端和接收端之间不插入语义通信模块，encoder 特征通过理想旁路直接送入接收端。先训练或稳定双头任务模型：
   - 检测头消费恢复前的语义特征；
   - 重建头学习从语义特征恢复视频帧。
   这一阶段的目标，是先获得“没有通信损失时，检测与重建联合最优能够达到的任务上界”。

2. 插入 MDVSC 的结构逼近阶段。
   在双头上界模型稳定之后，插入 MDVSC 相关模块，包括 latent adaptor、semantic encoder、semantic decoder 和 per-level refinement。此时以冻结任务主路径为主：
   - 优先冻结 RT-DETR 主干、hybrid encoder、decoder 与 detection head；
   - 优先冻结已经稳定的重建主干；
   - 只训练通信相关模块，用 feature loss 去逼近无通信上界。
   这一步建议先使用 identity 或无噪声信道，把通信结构误差先单独学掉。

3. 信道感知的联合通信训练。
   在结构逼近完成后，加入 AWGN 或其他目标信道，让 MDVSC 在真实信道条件下继续优化。此时仍然以通信模块和接收端轻量修复模块为主要可训练部分，必要时允许重建分支后段做有限适配，但不建议过早解冻 RT-DETR 主路径。

4. 分层解冻的整体微调。
   当通信模块在真实信道下已经稳定后，再做小学习率整体微调。推荐采用分层解冻顺序：
   - 先解冻接收端 refinement 与 reconstruction branch；
   - 再视情况解冻 RT-DETR decoder 与检测头；
   - 最后如确有必要，再小范围触碰 encoder 之前的部分。
   不推荐一开始就全模型同时解冻，以免切点统计、latent 分布和通信模块输入同时漂移。

这条训练路径的核心原则是：

- 先把双任务模型本身训练出来；
- 再让通信模型去逼近这个任务上界；
- 最后再整体协同优化。

## 对仓库结构的含义

当前仓库中的模块应这样理解：

- src/semantic_rtdetr/detector：RT-DETR 基线与切点提取。
- src/semantic_rtdetr/semantic_comm/codec.py：当前 codec 占位实现，以及未来项目版 MDVSC 语义编码与解码逻辑的落点。
- src/semantic_rtdetr/semantic_comm/channel.py：通信信道仿真与码率侧指标。
- src/semantic_rtdetr/pipeline：实验流程编排。

当前 codec 与 channel 模块的作用，是帮助架构稳定化。它们还不是最终的项目版 MDVSC 实现。

## V1 默认配置提案

除非后续实验明确推翻，第一版项目实现默认采用以下配置。

### 1. 多尺度传输策略

- 第一版正式实验默认传输全部 level，即 level 0、level 1、level 2 全部参与通信链路。
- 三个 level 虽然都传，但不采用相同的 latent 预算与稀疏预算。
- 默认优先级为：level 2 最高，level 1 次之，level 0 最低。

### 2. 每层 latent 维度

第一版默认 latent 维度设置为：

- level 0：256 -> 48
- level 1：256 -> 64
- level 2：256 -> 96

这组配置的意图是：

- 对高分辨率 level 0 强降维，优先控制总体传输成本；
- 对语义更强的 level 2 保留更高表示容量；
- 让三个 level 保持非对称预算分配，避免平均分配导致高分辨率层吞掉大部分码率。

### 3. 公共语义建模范围

- 第一版只做按 level 的公共语义建模，不做跨 level 的公共语义建模。
- 对每个 level 独立执行 temporal aggregation、common/individual 分解和后续传输。
- 跨 level 公共建模被视为后续增强项，而不是第一版默认结构。

### 4. Structured sparse 默认单元

第一版默认采用结构化稀疏，而不是逐元素稀疏。

- 公共分支：按单通道做弱稀疏，每个通道独立决定是否保留，不做分组。
- 个体分支：按空间块做结构化稀疏。

#### 公共分支默认设置

- 公共分支保留比例默认按 level 非对称设置：
   - level 0：保留 50% 的通道
   - level 1：保留 62.5% 的通道
   - level 2：保留 75% 的通道

也就是在默认 latent 维度下：

- level 0：48 通道中默认保留 24 个通道
- level 1：64 通道中默认保留 40 个通道
- level 2：96 通道中默认保留 72 个通道

这一默认值体现的是“公共语义小而稳、选择粒度细，但不做过强稀疏”的设计取向。

#### 个体分支默认设置

- 个体分支按空间块做稀疏，默认 block 大小为：
   - level 0：8x8
   - level 1：4x4
   - level 2：2x2

- 个体分支保留比例默认按 level 非对称设置：
   - level 0：保留 12.5% 的 block
   - level 1：保留 18.75% 的 block
   - level 2：保留 25% 的 block

这一默认值体现的是“个体残差适合更激进压缩，但低分辨率语义层应保留更多局部变化能力”的设计取向。

### 5. 重建头默认拓扑

- 第一版重建头默认采用 FPN/U-Net 风格的多尺度解码器。
- 恢复后的 level 2 作为主干语义起点，逐步上采样并融合 level 1 和 level 0。
- 第一版默认先做逐帧重建，不额外叠加重型时序重建头。

### 6. 损失与训练调度默认策略

- 第一版不采用“固定一组权重从头训到尾”的做法。
- 默认采用阶段式训练与阶段式损失调度：
   - 上界模型阶段：以检测和重建为主；
   - 结构逼近阶段：增强特征一致性约束，弱化码率约束；
   - 信道训练阶段：逐步提高码率和信道鲁棒性约束；
   - 整体微调阶段：按验证集结果平衡检测与重建权重。

## 仍需明确的开放设计项

在大规模训练开始前，以下工程决策仍需被明确：

- 上述 V1 默认配置是否需要在正式实验前先做一轮小规模敏感性验证；
- 公共语义是否最终只按 level 建模，还是在 V2 中引入跨 level 公共建模；
- structured sparse 默认单元在真实训练中是否需要从“公共分支按通道、个体分支按空间块”的配置扩展到联合单元；
- 重建头的具体拓扑；
- 检测、重建与码率损失的具体初始权重与调度曲线。
